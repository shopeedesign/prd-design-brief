# 面向设计师的 PRD 简报（prd-design-brief）

用于把 PRD 转换成设计师可直接开工的高密度执行摘要。这个 skill 的重点不是复述原文，而是帮助设计师快速建立业务认知、拆解设计单元、识别风险，并判断设计工作量和待确认问题。

## 核心能力

- 从设计师视角提炼 PRD，而不是按产品经理写法重复原文。
- 归并零散需求，输出更接近设计执行的结构化简报。
- 自动整理主流程，并根据复杂度选择更合适的流程呈现方式。
- 暴露设计机会点、需求缺口、边界不清和潜在风险。
- 支持按输出范围生成简报，而不是每次都输出全量章节。

## 适用场景

- 设计师拿到一份长 PRD，想快速建立全局理解并准备开工。
- 需要在 kickoff 前拆页面、拆模块、估设计工作量。
- 想把一份 PRD 压缩成适合设计团队同步的高密度简报。
- 用户提供 Confluence 链接、`pageId` 或 PRD 正文，并明确希望从设计角度理解需求。

## 不适用场景

- 用户只是想搜或定位 Confluence 文档，此时更适合 `confluence-search`。
- 用户要分析的是 BRD，而不是 PRD，此时更适合 `brd-analyzer`。
- 用户已经有设计稿，希望审查方案质量，此时更适合 `design-selfcheck`。
- 用户要撰写 PRD，而不是阅读和拆解 PRD。

## 输入与前置条件

- 支持三类输入：`Confluence 链接`、`Confluence pageId`、`直接粘贴 PRD 正文`。
- 如果输入来自 Confluence，skill 复用 `confluence-search` 的抓取能力与本地凭据。
- 本地需存在 `~/.confluence-credentials`，并包含：
  - `CONFLUENCE_BASE_URL`
  - `CONFLUENCE_AUTH_TYPE`
  - `CONFLUENCE_TOKEN`
- 凭据缺失时，应先提示补充，不绕过权限直接抓取。

## 使用方式

### 推荐指令

```text
请用 prd-design-brief 帮我从设计师视角看这份 PRD：https://...
```

```text
帮我做一份 PRD 简报，pageId=123456789，重点看页面范围和工作量。
```

```text
我是设计师，下面这段 PRD 帮我整理成可以直接开工的设计简报：
[粘贴正文]
```

### 输出范围选择

如果用户没有明确指定要看哪些部分，skill 会先确认输出范围。默认有 3 个预设包：

- `标准全量版`：适合首次开工或跨团队对齐。
- `开工重点版`：适合先抓主流程、页面范围和风险点。
- `洞察评审版`：适合方案评审和设计策略判断。

如果用户未选择但继续要求开始分析，默认按 `标准全量版` 输出。

## 工作流程

1. 获取 PRD 原文，来源可以是链接、`pageId` 或直接正文。
2. 如果用户尚未指定输出范围，先确认要使用哪一种输出包。
3. 通读 PRD，建立核心场景、角色、主流程和系统模块认知。
4. 将零散需求归并为设计单元，而不是逐条复述原文。
5. 识别设计机会点与 PRD 缺口，并把模糊点归入待明确问题。
6. 按所选输出范围生成简报，必要时标注“推断”或“行业经验”。

## 输出结果

可用章节池如下，实际输出会根据用户选择裁剪：

- `TL;DR`：用 1-2 句话说明需求目标、对象和整体量级。
- `需求背景`：为什么做、要达成什么、谁来用、范围和上下文。
- `业务流程`：用 ASCII 泳道图、纵向步骤流、水平管道流或场景卡片表达主链路。
- `功能点`：按设计模块归并，而非按 PRD 条目平铺。
- `前端页面`：按设计单元整理页面或模块改动范围。
- `设计机会点`：指出值得深做的高杠杆设计方向。
- `待明确问题`：指出 PRD 中的断点、矛盾、未闭环或边界不清之处。
- `设计工作量预估`：帮助判断需求复杂度与投入规模。
- `待确认问题`：沉淀需要和产品、研发或业务再确认的问题。

## 目录说明

- `SKILL.md`：skill 的完整触发规则、流程和输出格式。

## 注意事项

- 这个 skill 的目标是“帮助设计师快速开工”，不是生成 PRD 摘抄版。
- TL;DR 必须优先使用白话，避免未解释的业务缩写。
- 业务流程只保留主干链路，异常和分支应单独标出，不要把图画得过细。
- 任何基于常识补足的判断，都应明确标注为“推断”或“行业经验”。
