# prd-design-brief

面向设计师的 PRD 理解与简报 skill。

它的目标不是复述 PRD，而是把一份较长、较散的需求文档，压缩成一份设计师可以快速开工的执行摘要：看完后能知道为什么做、谁会用、主流程是什么、要设计哪些模块和页面、工作量大概落在哪些地方。

## 适用场景

当你遇到下面这些情况时，这个 skill 很适合：

- 你拿到一份 PRD，但想先从设计视角快速建立全局理解。
- 你要在开工前做需求拆解，想快速识别页面范围和设计重点。
- 你需要把长 PRD 压缩成一份更适合设计团队同步的简报。
- 你希望输出不是“产品复述”，而是“设计师能立刻进入执行状态”的版本。

## 它会在什么时候触发

以下任一情况都适合调用：

- 用户明确提到：`PRD 简报`、`设计简报`、`需求理解`、`PRD 摘要`。
- 用户给出 PRD 链接、`pageId` 或正文，并表达“从设计角度理解”的诉求。
- 用户是设计师，想知道这个需求要做哪些页面、哪些模块、工作量大不大。

不适合的场景：

- 只是想搜 Confluence 文档，而不是分析 PRD。
- 要分析的是 BRD，不是 PRD。
- 已经有设计稿，想做设计自查。
- 需要撰写 PRD，而不是阅读和拆解 PRD。

## 输入方式

这个 skill 支持三类输入：

### 1. Confluence 链接

适合已经有线上 PRD 页面时使用。

示例：

```text
请用 prd-design-brief 帮我看这个需求：https://xxx.atlassian.net/wiki/spaces/AAA/pages/123456789/...
```

### 2. Confluence pageId

适合你只拿到了页面 ID。

示例：

```text
帮我从设计视角做一份 PRD 简报，pageId=123456789
```

### 3. 直接粘贴 PRD 正文

适合没有链接、只有文档内容，或者不方便直接访问 Confluence 时使用。

示例：

```text
我是设计师，下面这段 PRD 帮我做成设计简报：
[粘贴正文]
```

## 输出内容

这个 skill 默认会输出一份中文简报，通常包含以下部分：

1. 一句话概述：用最短的话讲清楚需求目标、对象、量级。
2. 需求背景：为什么做、要达到什么、谁来用、范围边界、上下文系统。
3. 业务流程：把主链路整理成易读的流程表达。
4. 功能点：按设计模块聚合，而不是按 PRD 小条目平铺。
5. 前端页面：按“设计单元”归并页面改动，帮助快速估范围。
6. 设计工作量预估：帮助判断这是轻量改动还是中大型需求。
7. 待确认问题：把模糊点和风险点提前暴露出来。

## 这个 skill 的特点

- 站在设计师视角，而不是产品经理视角。
- 强调归并和抽象，不机械复述 PRD。
- 优先帮助建立“场景感”和“工作量感”。
- 会把零散需求整理成更接近设计执行的结构。
- 默认控制篇幅，让人能在几分钟内读完。

## 业务流程怎么呈现

这个 skill 不会固定只用一种流程格式，而是会根据需求复杂度自动选择更合适的表达方式：

- `ASCII 泳道图`：适合多角色协作、存在明显交接的流程。
- `纵向步骤流`：适合系统多、步骤多、泳道图会过宽的场景。
- `水平管道流`：适合流程简单、角色少的场景。
- `场景卡片`：适合多个独立使用场景并列的需求。

这样做的目的是让设计师先建立完整主线，再看功能点和页面拆分。

## 推荐提问方式

下面这些问法通常能得到比较稳定的结果：

```text
请用 prd-design-brief 帮我从设计师视角看这份 PRD。
```

```text
我是设计师，帮我把这个需求整理成一份可以直接开工的设计简报。
```

```text
这个 PRD 请帮我输出：背景、流程、功能模块、页面范围、工作量预估和待确认问题。
```

## 示例输出风格

下面是一个简化示意，不代表真实业务内容：

```md
一句话概述
本期要为跨境卖家新增直发表达能力，覆盖卖家发货、运营处理和物流状态流转，属于中等复杂度、多角色协同需求。

一、需求背景
- 当前卖家在特殊履约场景下缺少直发能力，导致处理链路长、人工介入多。
- 本期目标是缩短履约链路，同时让运营和卖家都能看到更清晰的状态。
- 核心用户包括运营、卖家和履约相关角色。

二、业务流程
Seller · Ops · OMS · 物流系统

1 · [Ops] 配置资格 → 卖家获得能力
2 · [Seller] 发起操作 → 系统生成任务
3 · [OMS] 分发状态 → 下游同步
4 · [物流] 回传结果 → 页面展示更新

设计关注点
- 卖家发起动作需要新的表单或抽屉。
- 状态流转较长，需要统一状态文案和反馈层级。

三、功能点
- 能力配置模块：负责资格开通与校验。
- 发起操作模块：负责创建、编辑、提交。
- 状态跟踪模块：负责状态展示、异常提示、回传结果。

四、前端页面
| 页面/设计单元 | 端 | 改造 or 新增 | 类型 | 关键交互/重点 | 说明 |
|---|---|---|---|---|---|
| 资格配置页 | B 端 | 改造 | 表单页 | 开通条件、校验反馈 | 运营侧 |
| 发起操作抽屉 | B 端 | 新增 | 抽屉 | 创建、编辑、提交 | 卖家侧 |
| 状态列表与详情 | B 端 | 改造 | 列表+详情 | 状态展示、异常提示 | 覆盖 2 个页面 |
```

## 使用前提

如果输入来源是 Confluence，这个 skill 依赖本地的 Confluence 凭据文件：

- `~/.confluence-credentials`

需要包含：

- `CONFLUENCE_BASE_URL`
- `CONFLUENCE_AUTH_TYPE`
- `CONFLUENCE_TOKEN`

如果凭据缺失，skill 应该停下来提示补充，而不是绕过权限继续抓取。
